Summary: Fix the reported medium-severity security findings with narrow, reusable hardening: add shared abuse controls for expensive endpoints, centralize GitHub URL/token sanitization in apps/web, tighten request validation and tar extraction, upgrade encryption to authenticated mode with legacy read support, and lock down the server-side fetch/clone paths in packages.

Context: The findings cluster into four concrete problem areas. (1) Expensive authenticated endpoints in [apps/web/app/api/chat/route.ts](#workspace-file=apps/web/app/api/chat/route.ts), [apps/web/app/api/generate-pr/route.ts](#workspace-file=apps/web/app/api/generate-pr/route.ts), and [apps/web/app/api/sandbox/route.ts](#workspace-file=apps/web/app/api/sandbox/route.ts) have no reusable rate-limit layer today. The repo already has Redis plumbing in [apps/web/lib/redis.ts](#workspace-file=apps/web/lib/redis.ts), but no abuse-control helper. (2) GitHub owner/repo parsing is inconsistent: validation helpers live in [apps/web/lib/github/repo-identifiers.ts](#workspace-file=apps/web/lib/github/repo-identifiers.ts), while weaker regex parsing exists in [apps/web/lib/github/client.ts](#workspace-file=apps/web/lib/github/client.ts) and [apps/web/lib/github/tarball.ts](#workspace-file=apps/web/lib/github/tarball.ts); [apps/web/app/api/sandbox/route.ts](#workspace-file=apps/web/app/api/sandbox/route.ts) validates one URL but still passes the raw user-controlled URL into sandbox creation. (3) Several paths log or return raw git/API errors that can include authenticated GitHub/Vercel URLs or sensitive token hints. (4) A few boundaries still trust unvalidated input directly: [apps/web/app/api/sessions/[sessionId]/route.ts](#workspace-file=apps/web/app/api/sessions/[sessionId]/route.ts), [apps/web/lib/crypto.ts](#workspace-file=apps/web/lib/crypto.ts), [apps/web/lib/github/tarball.ts](#workspace-file=apps/web/lib/github/tarball.ts), [packages/agent/tools/fetch.ts](#workspace-file=packages/agent/tools/fetch.ts).

System Impact: The system change is mostly at trust boundaries, not in core product flow. Source-of-truth for rate limiting becomes a user+scope window counter (Redis when configured, process-local fallback otherwise). Source-of-truth for GitHub repository coordinates becomes a single strict parser/canonicalizer in apps/web instead of per-file regexes. Encryption changes only affect ciphertext format at write time; decrypt will remain backward compatible for existing stored secrets. The main new API behavior is explicit 429 responses for abusive requests and stricter 400 responses for malformed GitHub/session inputs. Dependent flows that already use repo owner/name, sandbox cloning, auto-commit push, and tarball extraction will inherit the stricter validation and redaction paths.

Approach: Use the smallest coherent fix set: add one shared rate-limit helper for the three expensive apps/web endpoints; move GitHub URL/path/token safety into shared helpers in apps/web and consume them at every affected boundary; sanitize only the specific logging/error surfaces that leak secrets instead of broad logging refactors; use Zod only where mass assignment is the issue; and keep crypto backward compatible by reading legacy CBC ciphertext while writing authenticated GCM ciphertext.
